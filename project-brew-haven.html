02 — Responsive Web Design

Brew Haven
Coffee Co. Site

Designed and built a fully responsive ordering experience for a fictional café brand from scratch — grounded in persona-driven scenarios, information architecture principles, and a mobile-first approach.

Role UX Designer & Front-End Developer
Methods Persona Development, IA, Prototyping
Tools HTML, CSS, JavaScript, Figma
Context MSU — Web Design & Development
01 — The Brief

Design a café experience for real people, not just screens

The challenge: design and build a complete web presence for Brew Haven Coffee Co. — a fictional independent café — that serves three very different user types: the loyal regular ordering their usual, the first-time visitor exploring the menu, and the remote worker scoping out a place to spend the afternoon.

Rather than jumping straight to code, the project started with who would actually use the site and what they needed to accomplish — forcing every design decision through the lens of real user goals.

"The most important design decision on a café site isn't the color palette. It's whether someone can find the menu and the hours in under 10 seconds on their phone while walking down the street."

02 — Understanding the Users

Three personas, three distinct journeys

Before wireframing a single screen, I developed three user personas based on behaviors common to independent café customers — then mapped each persona to a primary scenario that would drive the information architecture.

  • P1
    The Regular — Maya, 24 Stops in 4x a week. Needs to see her go-to items fast, check seasonal specials, and find loyalty rewards info. Values speed above all else on mobile.
  • P2
    The Explorer — James, 34 First visit, referred by a friend. Wants to browse the full menu, understand the vibe, and confirm it's worth going out of his way for. Desktop or tablet.
  • P3
    The Worker — Priya, 29 Looking for a spot to set up for 3 hours. Needs to know about WiFi, outlets, noise level, and hours before she commits. Will Google it first, then go straight to the site.

Each persona revealed a different priority for the site hierarchy — which is why the homepage landing view needed to answer all three questions simultaneously without overwhelming any one user type.

03 — Information Architecture

Deciding what goes where — and why

With three personas and their scenarios mapped out, I drafted a site map and ran a quick card sort exercise with classmates to validate the navigation labels before wireframing. The result was a clean 5-section structure:

Home

Hero + Quick Access

Hours, location, and a direct link to the menu — the three things every persona needs within 5 seconds.

Menu

Browsable + Filterable

Full menu with category tabs (drinks, food, seasonal) and clear pricing. No PDFs — fully HTML for accessibility.

About

Story + Vibe

Brand story, photos, and the practical info remote workers need: WiFi ✓, outlets ✓, pet-friendly ✓.

Order

CTA-Focused

Online pre-order flow with pick-up time selection — streamlined for the regular who wants zero friction.

04 — Design & Build

Mobile-first, then scale up

All CSS was written mobile-first using a fluid grid with breakpoints at 480px, 768px, and 1200px. Typography scales with clamp() so the reading experience stays comfortable on any device without media query spaghetti.

  • 01
    Low-fi wireframes in Figma Rough layouts for all 5 pages across mobile and desktop, shared with peers for a quick hallway usability check before any code was written.
  • 02
    Design system first Defined a color palette, type scale, spacing scale, and component library (buttons, cards, nav states) in CSS variables before touching page layouts — preventing inconsistency at scale.
  • 03
    Built mobile-first in HTML/CSS/JS Semantic HTML throughout for accessibility. CSS Grid and Flexbox for layouts. Vanilla JS for the menu tab interaction and a smooth-scroll order flow.
  • 04
    Cross-device testing Tested on physical iOS and Android devices and across Chrome, Safari, and Firefox. Fixed 3 layout bugs found only on real devices that DevTools emulation missed.
