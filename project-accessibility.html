University Portal
Accessibility Audit

A systematic WCAG-based audit of a university web portal ‚Äî screen reader testing, color contrast analysis, empathy mapping, and developer-ready recommendations with clear severity tiers.

Role UX Researcher / Accessibility Analyst
Methods WCAG Audit, Screen Reader Testing, Empathy Mapping
Tools NVDA, axe DevTools, Figma, Color Contrast Analyzer
Context MSU ‚Äî Evaluating Human-Centered Technology
01 ‚Äî Why Accessibility

Inaccessible tools exclude real people

University portals are not optional ‚Äî students use them to register for classes, access grades, pay tuition, and communicate with advisors. For students who rely on assistive technology, a poorly built portal isn't just frustrating. It's a barrier to basic academic participation.

The goal of this audit was to move beyond assumptions about who uses accessibility features and conduct a rigorous, evidence-based evaluation of where the portal failed its users ‚Äî and produce recommendations clear enough for a developer to act on immediately.

"Accessibility isn't a feature. It's the foundation. Every barrier found in this audit is a real student who can't complete a real task."

02 ‚Äî Audit Methodology

Four-part evaluation framework

The audit combined automated scanning with manual testing ‚Äî because automated tools catch roughly 30‚Äì40% of accessibility issues. The rest require human judgment and real assistive technology.

  • 01
    Automated Scan with axe DevTools Ran axe on 12 key pages to generate a baseline issue list. Flagged 47 violations across missing alt text, invalid ARIA labels, and form input errors.
  • 02
    Screen Reader Testing (NVDA + Chrome) Navigated core user flows ‚Äî login, course registration, grade lookup ‚Äî using only a keyboard and NVDA screen reader. Documented every point of confusion or failure with timestamped notes.
  • 03
    Color Contrast Analysis Tested all text and interactive element color combinations against WCAG AA (4.5:1) and AAA (7:1) ratios using the Color Contrast Analyzer tool. Flagged 11 failing combinations on high-traffic pages.
  • 04
    Empathy Mapping Synthesized findings from all testing phases into an empathy map for a screen reader user attempting course registration ‚Äî surfacing emotional as well as functional pain points.
03 ‚Äî Key Findings

Where the portal fell short

47

Automated Violations

Found across 12 pages ‚Äî missing alt text and invalid ARIA roles were the most common categories.

11

Contrast Failures

Color combinations below WCAG AA threshold ‚Äî including the primary navigation links on the homepage.

0

Skip Navigation Links

No "skip to main content" links anywhere on the site ‚Äî keyboard users must tab through the entire nav on every page load.

3

Critical Flow Blockers

Course registration, password reset, and grade lookup each had at least one step completely inaccessible via keyboard alone.

The screen reader testing revealed a particularly painful issue: the course registration dropdown menus were built with custom JavaScript components that didn't expose their expanded/collapsed state to assistive technology. A screen reader user had no way to know whether clicking had done anything.

04 ‚Äî Recommendations

Developer-ready, prioritized by impact

All recommendations were organized into three tiers ‚Äî Critical (fix immediately, blocks access), High (fix in next sprint, degrades experience significantly), and Medium (fix within quarter, improves quality). Each included the specific page, the WCAG criterion violated, and the exact code change required.

05 ‚Äî Reflection

What this project changed for me

Before this audit, accessibility felt abstract. After spending hours navigating a university portal with only a keyboard and a screen reader, it became very concrete. I had to re-register for a class three times because the dropdown state wasn't announced. I couldn't complete a password reset without sighted help.

That experience fundamentally changed how I approach design. Accessibility considerations now come into every project from day one, not as a checklist at the end. Building accessible interfaces isn't harder ‚Äî it just requires thinking about a wider range of users earlier in the process.
